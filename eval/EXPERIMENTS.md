# DSF Evaluation Experiments

## Experiment 1: Network-Aware Scheduling

**Goal**: Show that network-aware scheduling (HEFT for ODAG, locality for CDAG) reduces makespan/latency compared to dependency-only random scheduling on heterogeneous edge links.

**Cluster**: anrg-3..6 with tc-shaped links:
```
anrg-3 ←1Gbps→ anrg-4
anrg-3 ←500Mbps→ anrg-5     anrg-4 ←500Mbps→ anrg-5
anrg-3 ←100Mbps→ anrg-6     anrg-4 ←100Mbps→ anrg-6     anrg-5 ←100Mbps→ anrg-6
```

### 1A: ODAG — Random vs HEFT ✅ COMPLETE

**Scenario**: IoBT Mission Snapshot (14 tasks, 5 layers, 100-200MB data transfers)
**Protocol**: 15 runs each of `scheduler: random` and `scheduler: heft`
**Metric**: Makespan (seconds)

**Results**:
- Random: mean 35.5s (std 5.5s, range 23-43s)
- HEFT: mean 25.8s (std 4.7s), **24.0s after convergence** (std 0.8s)
- **27% improvement overall, 32% after profiler converges (runs 4+)**
- Profiler converges in 3-4 runs via EMA smoothing

**Files**:
- Templates: `eval/network-aware/odag-iobt/template-{random,heft}.yml`
- Script: `eval/network-aware/run-odag-eval.sh`
- Results: `eval/network-aware/results/odag-makespan.csv`
- Profiler snapshots: `eval/network-aware/results/profiler-heft-run-*.json`
- Figures: `eval/network-aware/figures/odag-makespan-bar.png`, `odag-convergence.png`

**Run**:
```bash
./benchmarks/multi-odag-heft/setup-tc.sh          # apply tc shaping
kubectl apply -f eval/network-aware/odag-iobt/template-random.yml \
              -f eval/network-aware/odag-iobt/template-heft.yml
./eval/network-aware/run-odag-eval.sh 15
python3 eval/network-aware/plot-experiment-1.py
```

### 1B: CDAG — Random vs Locality ✅ COMPLETE

**Scenario**: Camera Fusion Pipeline (9 tasks, 5MB frames at 1Hz)
```
camera-1..4 → preprocess → detector → tracker → alert-sink + log-sink
```
**Protocol**: 5 instances × 120s each for random and locality schedulers
**Metrics**: End-to-end latency (camera→sink), throughput (msg/s)

**Results**:
- Random: 590ms avg latency, 1238ms p95 (high variance: 201-949ms)
- Locality: 462ms avg latency, 749ms p95 (more consistent)
- **22% avg latency improvement, 39% p95 tail latency improvement**

**Files**:
- Templates: `eval/network-aware/cdag-camera-pipeline/template-{random,locality}.yml`
- Task image: `eval/network-aware/cdag-camera-pipeline/task.py` + `Dockerfile`
- Script: `eval/network-aware/run-cdag-eval.sh`
- Results: `eval/network-aware/results/cdag-latency.csv`
- Figure: `eval/network-aware/figures/cdag-latency-throughput.png`

**Run**:
```bash
docker build -f eval/network-aware/cdag-camera-pipeline/Dockerfile \
  -t 192.168.1.163:5000/camera-pipeline:latest .
docker push 192.168.1.163:5000/camera-pipeline:latest
kubectl apply -f eval/network-aware/cdag-camera-pipeline/template-random.yml \
              -f eval/network-aware/cdag-camera-pipeline/template-locality.yml
./eval/network-aware/run-cdag-eval.sh 5 120
python3 eval/network-aware/plot-experiment-1.py
```

## Experiment 2: Scalability — P2P vs Centralized Communication

**Goal**: Show that DSF's P2P communication architecture scales better than centralized alternatives (NFS for ODAG, MQTT broker for CDAG) as fan-out width increases.

**Topology**: `source → worker-1..N → sink` with N = 2, 4, 6, 8

### 2A: ODAG — P2P data-agent vs NFS shared storage 📋 READY

**How P2P works**: Source writes locally, data-agent HTTP-pushes 50MB to each worker's node in parallel. Each transfer is independent.
**How NFS works**: Source writes to NFS server. ALL workers read from NFS. Workers write back to NFS. ALL I/O funnels through one pod.

**Metric**: Makespan per (transport × N), 5 runs each
**Expected**: NFS makespan grows faster with N (I/O bottleneck), P2P stays flat

**Infrastructure**:
- NFS server: `eval/scalability/nfs-server.yml` (Mosquitto on anrg-3)
- NFS mount: `eval/scalability/nfs-mount-daemonset.yml` (mounts on all workers)
- Controller: `cmd/odag-controller/main.go` mounts `/shared/dsf-outputs` in all ODAG pods
- Task: `eval/scalability/odag-task.py` (50MB payloads, works with both transports)
- Templates: generated by `eval/scalability/gen-odag-templates.py`
- NFS templates override `DSF_TRANSPORT_PATTERN=shared_volume` via task env vars

**Run**:
```bash
# Deploy NFS
kubectl apply -f eval/scalability/nfs-server.yml
kubectl apply -f eval/scalability/nfs-mount-daemonset.yml

# Rebuild odag-controller (adds /shared/dsf-outputs volume)
docker build -f cmd/odag-controller/Dockerfile -t 192.168.1.163:5000/odag-controller:latest .
docker push 192.168.1.163:5000/odag-controller:latest
kubectl rollout restart deployment/odag-controller -n dsf-system

# Build eval image
docker build -f eval/scalability/Dockerfile -t 192.168.1.163:5000/scalability-eval:latest .
docker push 192.168.1.163:5000/scalability-eval:latest

# Generate and apply templates
python3 eval/scalability/gen-odag-templates.py
kubectl apply -f eval/scalability/odag-templates/

# Run
./eval/scalability/run-odag-scalability.sh 5
```

### 2B: CDAG — ZMQ P2P vs MQTT broker 📋 READY

**How ZMQ works**: Source PUB → each worker SUB directly. No middleman.
**How MQTT works**: Source → Mosquitto broker → workers → broker → sink.

**Metrics**: Throughput at sink (msg/s), latency (p50/p95/p99)
**Parameters**: 100KB messages at 5 msg/s per source
**Expected**: ZMQ throughput scales linearly, MQTT saturates. ZMQ latency flat, MQTT grows.

**Infrastructure**:
- MQTT broker: `eval/scalability/mqtt-broker.yml`
- MQTT transport: `sdk/python/dsf_sdk/transport/mqtt.py`
- Task: `eval/scalability/task.py` (100KB streaming, parameterized)
- Templates: generated by `eval/scalability/gen-templates.py`

**Run**:
```bash
# Deploy MQTT broker
kubectl apply -f eval/scalability/mqtt-broker.yml

# Build eval image (same image as 2A)
docker build -f eval/scalability/Dockerfile -t 192.168.1.163:5000/scalability-eval:latest .
docker push 192.168.1.163:5000/scalability-eval:latest

# Generate and apply CDAG templates
python3 eval/scalability/gen-templates.py
kubectl apply -f eval/scalability/templates/

# Run
./eval/scalability/run-cdag-scalability.sh 120 3
```
